## Context

**当前状态**：
- 知识库系统已支持 PDF/Word/TXT/Markdown 文档的 ETL 管道（Apache Tika 解析 → 文本分块 → DashScope Embedding 向量化 → Pgvector 存储）
- 前端提供知识库管理面板，支持文件上传、状态查询、删除操作
- RAG 检索基于文本相似度（Cosine Distance），使用 1024 维向量

**约束条件**：
- 后端基于 Spring Boot + langchain4j 1.3.0-beta9 框架
- 向量存储使用 PostgreSQL + Pgvector 扩展
- Embedding 服务依赖 DashScope API（text-embedding-v3）
- 前端 Vue 3 + TypeScript 技术栈
- 文件上传通过 OSS 中转，后端接收公开访问 URL

## Goals / Non-Goals

**Goals：**
- 扩展知识库支持图片格式（JPG/PNG/WebP）的上传与向量化
- 使用 DashScope 多模态 Embedding 模型将图片直接编码为 1024 维向量，保留完整视觉特征
- 实现跨模态语义检索（文搜图、图搜文、文搜文）
- 保持与现有文本 ETL 管道的架构一致性（统一向量空间）
- 提供前端友好的图片上传体验（类型过滤、预览、进度反馈）

**Non-Goals：**
- ❌ 不实现实时视频流处理（当前版本仅支持静态图片）
- ❌ 不实现图片编辑或裁剪功能
- ❌ 不改变现有文本文档的处理流程
- ❌ 不实现混合向量方案（文本+图像双通道）

## Decisions

### 决策 1：图片向量化策略 - 使用 DashScope 多模态 Embedding API

**选择方案**：调用 DashScope Multimodal-Embedding API（`tongyi-embedding-vision-plus-2026-03-06`），通过 `parameters.dimensions: 1024` 强制输出 **1024 维向量**

**备选方案对比**：
| 方案 | 优点 | 缺点 |
|------|------|------|
| **A. 多模态 Embedding (CLIP)** ✅ | 保留完整视觉特征、统一向量空间、原生支持跨模态检索、成本低（¥0.0005/千token）、速度快（<1秒） | 需新增 API 调用逻辑 |
| B. Image→Text→Vector（VL+Embedding） | 复用现有管道 | 丢失视觉信息、成本高（¥0.2/次）、延迟大（5-15秒） |
| C. 混合向量（文本+图像） | 信息最全 | 架构复杂度高、检索逻辑需改造 |

**理由**：
1. **维度完全兼容**：通过 `parameters.dimensions=1024` 参数强制多模态模型输出 **1024 维**，与 `text-embedding-v3` **完全一致**
2. **统一语义空间**：官方文档明确说明"所有模态（文本、图片、视频）的向量均位于同一语义空间"，可直接进行余弦相似度计算
3. **无需数据库 Schema 变更**：现有 `document_chunks.embedding` 字段已是 `vector(1024)`，无需任何迁移脚本
4. **成本极低**：¥0.0005/千Token，比 VL API（¥0.2/次）便宜 **400 倍**
5. **性能优秀**：单次 API 调用 <1秒（vs VL 的 5-15秒）
6. **原生跨模态支持**：官方保证文本和图片向量可在同一空间比较，实现"文搜图"、"图搜文"

**技术实现要点**：
- 使用 **独立向量模式**（Independent Vectors）：为每个输入（文本/图片）分别生成独立向量
- 图片输入支持 **公开 URL 或 Base64 编码**两种格式
- 可选参数：
  - `dimensions: 1024` （必须与 text-embedding-v3 一致，通过 parameters.dimensions 控制）
  - `instruct: "用于RAG检索的图片内容描述"` （可选，提升检索精度 1-5%）
- API Endpoint：`POST https://dashscope.aliyuncs.com/api/v1/services/embeddings/multimodal-embedding/multimodal-embedding`

### 决策 2：图片分块策略 - 单 Chunk 模式（整张图片为一个向量单元）

**选择方案**：每张图片生成 **1 个 chunk**（chunk_index=0），存储完整图片向量

**备选方案对比**：
| 方案 | 优点 | 缺点 |
|------|------|------|
| **A. 单 Chunk 模式** ✅ | 简单高效、保留全局视觉特征、降低存储成本 | 无法对图片局部区域进行精细检索 |
| B. 多 Chunk 切割（象限分割） | 支持局部检索 | 破坏图片整体语义、增加复杂度 |
| C. VL 文本描述 + 分块 | 可按主题分割 | 依赖文本质量、丢失视觉信息 |

**理由**：
1. 多模态 Embedding 已捕获图片的**全局视觉特征**，无需再人为切割
2. 避免过度分割导致语义完整性破坏（如一张流程图被切成4块后每块都失去上下文）
3. 降低存储成本和检索复杂度（1张图 = 1个向量 vs 原方案的 1-10个向量）
4. 符合"所见即所得"的用户预期：用户上传一张图，就检索这一张图

**特殊情况处理（未来优化）**：
- 若后续发现超大图片（>4096x4096 或 >5MB）检索效果不佳，可考虑象限分割方案
- 当前版本暂不实现，通过配置预留扩展点

### 决策 4：前端实现 - 复用现有上传组件并增强

**选择方案**：在现有文件选择器中添加 `accept="image/*"` 过滤器，增加图片缩略图预览

**改动点**：
- `KnowledgePanel.vue` 的 `<input type="file">` 增加 MIME 类型提示
- 新增 `<ImagePreview>` 子组件，上传前显示缩略图（URL.createObjectURL）
- 文档列表中图片类型文档显示缩略图而非文件名

**理由**：
- 最小化前端改动，保持 UI 一致性
- 利用浏览器原生能力（FileReader）避免后端压力

## Risks / Trade-offs

**[API 兼容性风险] 多模态 Embedding 模型稳定性**
- **影响**：`tongyi-embedding-vision-plus-2026-03-06` 为 2026年3月快照版本，可能存在未知边界情况
- **缓解措施**：
  - 优先使用稳定版模型（如 `tongyi-embedding-vision-plus` 无日期后缀版本）
  - 添加完善的单元测试和集成测试覆盖各种图片格式
  - 实现降级方案：若多模态 Embedding 失败，可回退到 VL + 文本 Embedding 方案（配置开关控制）

**[成本风险] 多模态 Embedding 调用费用**
- **影响**：虽然单价低（¥0.0005/千token），但批量上传大量图片时仍可能产生费用
- **缓解措施**：
  - 配置开关控制是否启用图片上传功能
  - 添加用量统计和监控面板
  - 实现基于内容哈希的缓存机制（相同图片不重复调用 API）
  - 设置单次上传数量限制（建议 ≤ 10 张）

**[质量风险] 图片向量化效果依赖图片质量**
- **影响**：模糊、低分辨率、过度压缩的图片可能导致向量语义不准确
- **缓解措施**：
  - 前端校验图片分辨率（建议 ≥ 800x600）并给出警告提示
  - 限制单张图片大小（≤ 5MB，符合 DashScope API 要求）
  - ETL 失败时记录详细错误信息，支持手动重试
  - 提供图片预处理建议（如"请上传清晰度较高的图片以获得更好检索效果"）

**[向量空间一致性风险] 文本与图片向量的语义对齐**
- **影响**：虽然官方保证同一语义空间，但实际检索中文本查询与图片结果的相似度分布可能与文本文档不同
- **缓解措施**：
  - 初期采用统一阈值策略（maxDist=0.5），后续根据用户反馈调整
  - 在 RAG 检索结果中明确标注来源类型（text/image），便于用户判断相关性
  - 收集 bad case 数据集，用于优化 instruct 参数或切换模型版本

**[存储风险] 向量数据增长**
- **影响**：每张图片新增 1 个 chunk（1024维 float32 = 4KB），相比原方案（1-10个文本chunk）反而更节省空间
- **积极影响**：
  - 单 Chunk 策略大幅降低存储占用（平均每图仅 4KB vs 原方案的 4-40KB）
  - 减少 Pgvector 索引维护开销
  - 提升检索速度（更少的向量需要比较）

## Migration Plan

**部署步骤**：
1. 数据库迁移：`knowledge_documents` 表新增 `content_type ENUM('text', 'image')` 字段，默认 'text'（**无需修改 document_chunks 表结构，embedding 字段已是 vector(1024)**）
2. 后端部署：发布包含 `MultimodalEmbeddingService` 的新版本
3. 配置更新：在 `application.yaml` 中添加多模态 Embedding 相关配置项
4. 前端部署：更新 KnowledgePanel 组件（图片上传、预览、状态展示）
5. 功能验证：
   - 上传测试图片（JPG/PNG/WebP），验证 ETL 全流程（<3秒完成）
   - 测试跨模态检索：用文本查询检索到图片结果，反之亦然
   - 验证混合知识库场景（文本+图片文档共存时的检索效果）

**回滚策略**：
- 若多模态 Embedding 出现问题，可通过配置 `knowledge-base.multimodal-embedding.enabled=false` 禁用功能
- 已上传的图片文档可正常删除，不影响文本文档使用
- 可选降级方案：临时切换回 VL + 文本 Embedding 模式（需额外开发，当前版本不实现）

## Open Questions

1. **是否需要支持视频向量化？** - DashScope API 已支持视频输入，但当前版本仅聚焦静态图片，后续可扩展
2. **是否需要对图片进行敏感信息检测？** - 企业场景可能涉及隐私保护（如身份证、合同截图），需评估合规要求
3. **instruct 参数如何优化？** - 初期使用通用指令"用于RAG检索的图片内容描述"，后期可根据业务场景定制（如技术文档类 vs 营销海报类）
4. **是否需要支持图片批量上传压缩包（ZIP）？** - 当前设计仅支持单张或少量批量上传，若需求强烈可后续迭代
