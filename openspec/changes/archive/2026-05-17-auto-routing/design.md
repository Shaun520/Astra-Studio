## Context

### 当前状态
系统已实现 `model-selection` 功能，支持用户在 GLM-5、DeepSeek V4 Flash、Qwen 3.6 Flash 三个模型间手动选择。`AiCodeHelperServiceFactory` 通过 `ConcurrentHashMap` 缓存 `(deepThink, webSearch, model)` 三维组合的服务实例。模型名称通过 `@RequestParam` 从前端传入，Factory 层仅做白名单校验后直接使用。

**核心问题**：用户缺乏模型专业知识，往往无法做出最优选择，导致：
1. **成本浪费**：用 GLM-5（¥16/百万输出Token）处理代码任务，而 DeepSeek（¥2/百万）更合适
2. **质量次优**：用 DeepSeek 处理需要中文文化背景的问答，而 Qwen 更擅长
3. **操作负担**：每次对话前都需要思考"该选哪个模型"

### 约束条件
- **向后兼容**：必须保留手动选择功能，`model=auto` 仅作为新增选项
- **性能要求**：分类延迟 < 10ms（不影响整体P99响应时间）
- **可解释性**：每次路由决策必须记录原因和置信度，支持调试
- **可维护性**：规则必须配置化（YAML），避免硬编码
- **无外部依赖**：不引入 ML 库或 NLP 服务（Phase 1）

### 利益相关者
- **终端用户**：希望零操作成本获得最优模型匹配
- **运维人员**：需要监控路由效果、调整分类规则
- **产品经理**：关注成本降低指标和用户满意度
- **开发团队**：需要清晰的架构便于扩展和维护

## Goals / Non-Goals

**Goals:**
- 实现基于用户输入文本的自动意图识别（准确率 ≥ 85%）
- 在 Controller 层集成自动路由逻辑（当 model 参数为 "auto" 时触发）
- 提供路由统计接口用于监控和调试（`/api/ai/routing-stats`）
- 支持用户手动覆盖自动决策（显式传 model 参数时优先）
- 完整的路由决策日志和置信度评估机制

**Non-Goals:**
- 不实现深度学习分类器（Phase 1 使用规则引擎，准确率目标 80-90%）
- 不做实时在线学习或用户反馈闭环（Phase 2 再考虑）
- 不修改前端核心交互流程（渐进式增强，可选展示路由信息）
- 不做 A/B 测试框架或多策略对比（后续迭代）
- 不处理多轮对话上下文感知（仅分析当前单条消息）

## Decisions

### Decision 1: 触发方式 - 显式 opt-in vs 全局默认

**选择**：显式 opt-in（用户传 `model=auto` 时启用自动路由）

**理由**：
- ✅ 向后兼容性最佳：旧客户端不传 model 或传具体值时行为完全不变
- ✅ 用户控制权保留：高级用户可随时关闭自动路由
- ✅ A/B 测试友好：可对比自动路由 vs 手动选择的实际效果
- ✅ 降低风险：即使分类器有 bug 也只影响 opt-in 用户

**替代方案考虑**：
- ❌ **全局默认（model 参数可选）**：所有请求都走自动路由
  - 风险高：分类器误判会影响所有用户
  - 回滚复杂：需要重新部署才能关闭
- ❌ **智能混合（80% 自动 + 20% 手动）**：随机分配
  - 实现复杂度增加（需要随机数生成和状态管理）
  -用户体验不一致（相同输入可能得到不同结果）

**未来演进路径**（如果 Phase 1 数据验证效果好）：
- Phase 2: 前端默认选中 "🤖 自动"，但保留手动选项
- Phase 3: 移除必填校验，model 参数完全可选（全局自动路由）

---

### Decision 2: 分类器技术选型 - 关键词 vs NLP API vs 本地 ML 模型

**选择**：关键词 + 正则表达式规则引擎（Rule-Based Classifier）

**理由**：

| 维度 | 关键词规则 | NLP API (阿里云) | 本地 ML (FastText) |
|------|-----------|------------------|-------------------|
| **延迟** | < 5ms ✅ | 50-200ms ❌ | 10-30ms ⚠️ |
| **准确率** | 80-90% ⚠️ | 95-98% ✅ | 92-96% ✅ |
| **依赖** | 无 ✅ | 网络调用 ❌ | JAR包 (~100MB) ⚠️ |
| **可解释性** | 强（每条规则可见）✅ | 黑盒 ❌ | 弱 ⚠️ |
| **维护成本** | 低（编辑YAML）✅ | 中（调参）⚠️ | 高（训练数据）❌ |
| **实现周期** | 1-2天 ✅ | 3-5天 ⚠️ | 1-2周 ❌ |

**关键决策点**：
1. **性能优先**：我们的场景对延迟敏感（实时对话），< 5ms 的分类时间可以忽略不计
2. **零依赖**：不需要网络调用或额外的模型文件，部署简单
3. **快速迭代**：修改 YAML 规则即可生效，无需重新训练或部署
4. **足够好**：对于三类意图（代码/中文/通用），规则引擎能达到 85%+ 准确率

**替代方案考虑**：
- ❌ **预训练 BERT 模型**：准确率高但推理慢（>100ms），且需要 GPU 或大型 CPU
- ❌ **调用外部 NLP API**：增加网络延迟和费用，且有服务可用性风险
- ❌ **传统机器学习（Naive Bayes/SVM）**：需要标注数据集，特征工程复杂

**未来演进路径**（当日请求量 > 10000 且需要更高准确率时）：
- Phase 2: 引入轻量级 FastText 模型（二进制文件 ~10MB，推理 < 20ms）
- Phase 3: 接入在线学习，根据用户反馈自动调整权重

---

### Decision 3: 规则定义格式 - YAML 配置 vs Java 注解 vs 数据库

**选择**：YAML 配置文件（`intent-rules.yaml`）

**理由**：

```yaml
# 示例规则结构（清晰易读）
intents:
  CODE_GENERATION:
    keywords:
      - ["写代码", "编程", "code", "implement", "debug", "修复bug"]
      - ["Python", "Java", "JavaScript", "算法", "数据结构"]
    patterns:
      - "帮我写.*(函数|方法|类|脚本)"
      - "如何实现.*"
    weight: 0.9
    target_model: "deepseek-v4-flash"
```

- ✅ **声明式语法**：非程序员也能理解和修改规则
- ✅ **热更新支持**：可通过 `@Scheduled` 任务定期检查文件变化并重载
- ✅ **版本控制友好**：规则变更可在 Git 中追踪
- ✅ **多环境适配**：dev/test/prod 可使用不同规则文件

**替代方案考虑**：
- ❌ **Java 注解（@IntentRule）**：编译期绑定，修改需重新打包部署
- ❌ **数据库存储**：过度设计，增加 DB 依赖和查询开销
- ❌ **硬编码在 Java 类中**：违反开闭原则，每次修改都要改代码

---

### Decision 4: 置信度阈值设定 - 多少算"足够确定"？

**选择**：0.6（60% 置信度阈值）

**计算公式说明**：
```
confidence = max(keyword_score, pattern_score) × intent_weight

其中：
- keyword_score = 匹配到的关键词组数 / 总关键词组数（0-1）
- pattern_score = 是否匹配到任一模式（0 或 1）
- intent_weight = 该意图的预设权重（0.7-0.9）
```

**示例计算**：
- 输入："帮我写一个 Python 快速排序"
  - CODE_GENERATION 关键词命中：["写"] + ["Python", "算法"] → 2/2 组 → score=1.0
  - 最终 confidence = 1.0 × 0.9 = **0.9** ✅ 高于阈值，路由到 DeepSeek

- 输入："今天天气怎么样"
  - 所有意图关键词都未命中 → keyword_score=0
  - 无模式匹配 → pattern_score=0
  - GENERAL_CHAT 兜底 → confidence = 0 × 0.7 = **0** ❌ 低于阈值，fallback 到 glm-5

**为什么选择 0.6**：
1. **经验值**：基于原型测试，正常匹配的 confidence 通常在 0.7-0.95 区间
2. **安全边际**：0.6 以下通常意味着无明确信号（如闲聊、模糊查询）
3. **兜底策略合理**：低置信度时使用通用模型（GLM-5），不会造成严重后果
4. **可调节性**：作为配置项，后续可根据实际数据动态调整

**替代方案考虑**：
- ❌ **固定 0.8（高阈值）**：过于严格，大量合法请求会被 fallback
- ❌ **动态阈值（基于历史分布）**：实现复杂度高，初期数据不足难以建模

---

### Decision 5: 统计数据存储 - 内存 vs Redis vs 数据库

**选择**：内存存储（ConcurrentHashMap + Atomic 变量）

**理由**：
- ✅ **零延迟**：统计数据读写都在内存中完成（< 0.1ms）
- ✅ **线程安全**：Java 并发包提供的原子操作保证一致性
- ✅ **简单可靠**：无需外部依赖，重启后清空（符合监控数据的特性）
- ✅ **足够用**：统计维度有限（约 10 个计数器），内存占用 < 1KB

**数据结构设计**：
```java
public class RoutingStats {
    private final AtomicLong totalRequests = new AtomicLong(0);
    private final AtomicLong autoRoutedCount = new AtomicLong(0);
    private final AtomicLong manualOverrideCount = new AtomicLong(0);
    private final AtomicLong fallbackCount = new AtomicLong(0);

    private final ConcurrentHashMap<String, AtomicLong> intentDistribution = new ConcurrentHashMap<>();
    private final ConcurrentHashMap<String, AtomicLong> modelUsage = new ConcurrentHashMap<>();

    private final DoubleAdder totalConfidence = new DoubleAdder(); // 用于计算平均值
}
```

**替代方案考虑**：
- ❌ **Redis**：适合分布式场景，但单实例应用过度设计
- ❌ **数据库（MySQL）**：写入延迟高（> 10ms），不适合高频更新
- ❌ **Prometheus + Grafana**：适合长期趋势分析，但实现复杂度高（需要 exporter）

**未来演进路径**（如果需要持久化或跨实例共享）：
- Phase 2: 定时将内存统计数据刷写到数据库（每分钟一次）
- Phase 3: 接入 Prometheus metrics，配合 Grafana 仪表盘

## Risks / Trade-offs

### Risk 1: 分类器误判导致用户体验下降

**描述**：用户问"什么是快速排序？"（概念性问题）→ 被错误路由到 DeepSeek（代码模型）→ 回答过于技术化，不够通俗易懂

**概率**：中等（估计 10-15% 的边界案例可能误判）

**影响**：高（用户可能认为 AI 能力下降）

**缓解措施**：
- ✅ **置信度阈值过滤**：< 0.6 时 fallback 到通用模型（GLM-5）
- ✅ **用户覆盖机制**：前端显示路由信息，用户可手动切换模型
- ✅ **持续优化规则**：收集负面反馈案例，每周调整关键词列表
- ✅ **透明日志**：记录每次路由的完整决策过程，便于快速定位问题

**权衡**：接受初期 10-15% 的误判率，换取快速上线和低成本维护；通过用户反馈逐步优化至 < 5%

---

### Risk 2: 规则维护成本随时间增长

**描述**：随着业务发展，新的任务类型涌现（如图片生成、视频处理），需要不断扩充规则库

**概率**：高（必然发生）

**影响**：中（需要定期投入人力维护）

**缓解措施**：
- ✅ **模块化规则结构**：每个意图独立配置，互不影响
- ✅ **文档化最佳实践**：编写《规则编写指南》，降低上手门槛
- ✅ **自动化测试**：为规则编写单元测试，防止回归
- ✅ **监控规则覆盖率**：统计 fallback 到通用模型的比例，发现缺失的意图类别

**权衡**：初期手工维护成本低（< 2小时/周）；当意图类型 > 10 个时再考虑引入 ML 辅助

---

### Risk 3: 性能开销（分类延迟影响整体响应时间）

**描述**：意图分类步骤增加了请求处理的额外延迟

**概率**：低（规则引擎本身很快）

**影响**：低（预计增加 2-5ms）

**缓解措施**：
- ✅ **预编译正则表达式**：启动时编译，运行时复用
- ✅ **短路优化**：一旦某意图达到高置信度（> 0.9）立即返回，不再计算其他意图
- ✅ **缓存近期结果**：对相同或相似的文本（如连续追问）可缓存分类结果（TTL=60s）
- ✅ **性能基准测试**：上线前进行压力测试，确保 P99 延迟 < 10ms

**权衡**：2-5ms 的延迟增量相比整体响应时间（通常 1-5s）可忽略不计；若未来成为瓶颈，可升级为本地 ML 模型

---

### Risk 4: 用户失去控制感（"AI 替我做决定"的抵触心理）

**描述**：部分高级用户可能不信任自动路由，希望能完全掌控模型选择

**概率**：中（取决于用户群体）

**影响**：中（可能导致功能使用率低）

**缓解措施**：
- ✅ **Opt-in 设计**：默认关闭，用户主动选择才启用
- ✅ **透明化决策**：前端显示 "已为您选择 DeepSeek（因为检测到代码相关关键词）"
- ✅ **一键覆盖**：路由信息旁提供切换按钮，允许即时更换模型
- ✅ **设置记忆**：记住用户的偏好（如某用户总是手动选 Qwen，则降低自动路由推荐权重）

**权衡**：平衡自动化便利性与用户自主权；通过 A/B 测试找到最佳平衡点（目标：70%+ 用户愿意使用自动路由）

---

## Migration Plan

### 部署步骤

1. **阶段一：灰度发布（第 1 周）**
   - 仅对内部测试账号开放 `model=auto` 选项
   - 收集路由日志和用户反馈
   - 监控关键指标（命中率、置信度分布、手动切换率）

2. **阶段二：小范围公测（第 2 周）**
   - 对 10% 的真实用户开放自动路由
   - 前端 UI 新增"🤖 智能"选项（置于模型列表顶部）
   - 收集更大规模的数据样本

3. **阶段三：全量发布（第 3 周）**
   - 所有用户均可使用自动路由
   - 根据前两周数据微调规则和阈值
   - 编写用户帮助文档和 FAQ

### 回滚策略

- **即时回滚**：通过配置中心将 `auto-routing.enabled` 设为 `false`，所有请求立即回退到手动模式
- **代码回滚**：若出现严重 Bug，移除 AutoRoutingService 调用即可（向后兼容，不影响现有功能）
- **数据清理**：回滚后内存中的统计数据会在下次重启时自动清空

### 监控告警

- **路由命中率 < 50%**：警告级别（可能规则失效）
- **Fallback 率 > 30%**：警告级别（置信度阈值可能过高）
- **平均置信度 < 0.7**：信息级别（建议审查规则质量）
- **分类延迟 P99 > 20ms**：严重级别（性能异常）

## Open Questions

1. **是否需要支持多语言意图识别？**
   - 当前规则主要针对中文和英文混合场景
   - 如果有日文/韩文用户需求，需要扩展关键词库
   - **决策时机**：Phase 2（根据用户反馈决定）

2. **是否需要记录用户反馈（thumbs up/down）？**
   - 可用于优化规则权重和识别误判案例
   - 但会增加前端改造工作量
   - **决策时机**：Phase 2（先验证基本功能价值）

3. **是否需要支持上下文感知路由？**
   - 当前仅分析单条消息，不考虑对话历史
   - 例如：用户先问"什么是快速排序"（概念→Qwen），再问"帮我写一个"（代码→DeepSeek）
   - **决策时机**：Phase 3（需要对话历史存储和状态管理）
